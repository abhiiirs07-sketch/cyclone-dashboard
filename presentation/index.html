India's eastern coastline faces some of the world's most intense tropical cyclones. Disaster managers lack unified, real-time satellite intelligence to coordinate rescue operations, assess flood extents, map hazard zones, and generate automated impact reports during the critical 72-hour response window.

Meteorological, flood, vegetation, and population data exist in separate siloed systems with no unified view.

Slow Response

📊

No Automation

Impact reports require manual field surveys, delaying NDRF operations and relief fund allocation decisions.

Slide 07 — Google Earth Engine
Petabyte-Scale Satellite Analysis

🛰️ Satellite Data Sources

  • Sentinel-1 SAR — Flood detection through cloud cover using radar backscatter
  • Sentinel-2 Optical — High-resolution NDVI for vegetation damage mapping
  • ERA5 Reanalysis — Hourly meteorological variables (wind, temp, humidity)
  • GPM IMERG — Near-real-time global precipitation measurement
  • WorldPop — 100m resolution population density grids
  • MODIS Land Cover — Annual LULC classification at 500m resolution

⚙️ GEE Processing Pipeline

  • Pre/Post Image Differencing — Change detection for flood and vegetation loss
  • SAR Threshold Analysis — Automated flood extent classification
  • Time-Series Reduction — Temporal compositing for meteorological trends
  • Zonal Statistics — District-level aggregation for administrative reports
  • Multi-criteria Overlay — Weighted composite hazard index scoring
  • Dynamic Tile Generation — XYZ tile URLs for interactive map rendering

🔧 Authentication & API

  • OAuth 2.0 personal authentication for local development
  • Service Account credentials for production deployment
  • Refresh token-based auth for seamless server operation
  • Project ID: jarvice-ng
  • Python earthengine-api v1.1+ with lazy initialization
  • Graceful error handling — "not configured" vs crash

⚡ Performance Optimizations

  • Pre-computed Cache — JSON cache files for all 12 modules (instant load)
  • Lazy GEE Init — Earth Engine only initializes on first API call
  • Tile URL Caching — Map tile URLs cached to avoid repeated GEE calls
  • Async FastAPI — Non-blocking I/O for concurrent requests
  • CORS Optimization — Preflight caching for cross-origin tile requests

Slide 09 — Deployment Architecture
Fully Online, Accessible from Anywhere

🌐 Frontend — Netlify

Next.js 15 static export deployed to Netlify's global CDN. Accessible from any browser worldwide with zero downtime.

prismatic-vacherin-06ce2d.netlify.app

🚂 Backend — Railway

Python FastAPI server auto-deployed from GitHub. Handles GEE API calls, caching, and 12 analysis module endpoints.

cyclone-dashboard-production.up.railway.app

🐙 Source — GitHub

Full codebase version-controlled. Push to main triggers automatic redeploys on both Netlify and Railway.

github.com/abhiiirs07-sketch/cyclone-dashboard

🔐 Auth — Google OAuth 2.0

Earth Engine authenticated via refresh token stored as Railway environment variable. No manual re-authentication needed.

⚡ Caching Strategy

Pre-computed JSON cache files for Cyclone FANI ensure instant response times without waiting for GEE computation.

🛡️ Security

CORS restricted to known frontend URLs. Credentials stored as environment variables, never in source code.
